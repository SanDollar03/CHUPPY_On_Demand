# ondemand専用プロジェクト 引き継ぎメモ

このプロジェクトは **ondemand専用の最小構成** です。
手動画面 / 自動画面 / ナレッジ画面は削除済みで、**ondemand.html 関連だけを残しています**。

## 最新コード
新規チャットには、まず最新版 ZIP をアップロードしてください。
- `ondemand_only_project_py_split_refactor.zip`

```text
ondemand_only_project/
├─ app.py
├─ ondemand_core.py
├─ ondemand_queue.py
├─ prompt.md
├─ templates/
│  └─ ondemand.html
└─ static/

   └─ style.css
```

## プロジェクトの目的
ユーザーが自分の担当フォルダにファイルを追加すると、
公平に分散したキューで順番待ちさせながら、
**Markdown変換** を自動実行する ondemand 専用アプリです。
Difyナレッジ登録は行わず、保存先フォルダへのMarkdownファイル保存で完了します。

## 重要な作業ルール
- 機能変更は慎重に行う
- 既存動作を壊さず、必要な修正だけを入れる
- **変更時は完全ファイルで返す**
- **Goと言うまで記憶してください**
- **Goと言うまでコメントアウトは不要**

## 現在の主要仕様
1. 画面は `ondemand.html` のみ
2. `app.py` 起動時から常時フォルダ監視
3. 差分ファイルを検出したらキューへ投入
4. キューはフォルダ公平性を保つラウンドロビン方式
5. 順番が来たファイルは Markdown 変換して保存先フォルダへ出力
6. 変換スタイルは RAG向け Markdown
7. chunk separator は `***`
8. Difyナレッジ登録は行わない
9. 失敗時は最大5回リトライ
10. 最終的にエラーになったものは **元ファイル / Markdown を自動削除**
11. 内容一覧からの削除でも **元ファイル / Markdown / 同一キュー** を削除
12. キューのエラー行から **手動削除**（元ファイル / Markdown 削除）または **手動再試行**（Markdown を削除してキュー最後尾へ再投入）が可能
13. 保存先に同名Markdownが既に存在する場合は差分なしとしてスキップ
14. 操作ログ `ondemand_queue.log` は軽量形式
   - `timestamp`
   - `folder_name`
   - `file_name`
   - `status`

## ナレッジ名ルール
Difyナレッジ登録はしないためナレッジ名は使用しません。

## Markdown保存先ルール
変換したMarkdownファイルは、タイムスタンプ（変換時刻yyyymmddhhmmss形式）をプレフィックスとしたファイル名で保存されます。

例:
`\\172.27.23.54\disk1\Chuppy\P\PJ\PJ4\元データ\生産\1.pdf`
→
`\\172.27.23.54\disk1\Chuppy\P\PJ\PJ4\マークダウン形式\生産\20260430143525_1.md`

フォーマット: `{変換時刻yyyymmddhhmmss}_{元ファイル名}.md`

差分判定は、保存先ディレクトリ内で `*_{元ファイル名}.md` パターンのファイルが存在するかチェックします。

## 現在のUI仕様・実装済みポイント
- 固定ヘッダー + 左固定サイドバー
- ondemand画面のみ
- 左ツリーはかなりコンパクト寄りのレイアウト
- フォルダツリーは **Lv1展開済みが初期状態**
- ツリーは **Lv4の `元データ` を見せず**、見た目上は Lv3 から Lv5 を選ぶ形
- ツリーの展開記号は **展開前 `>` / 展開後 `∨`**
- フォルダ名テキストをクリックしても展開/閉じる操作が可能
- ツリー右側に **配下対応ファイル数** を `人材育成(30)` のように表示
- ファイル数の文字色は赤
- Lv5 フォルダ名の前に絵文字を付与
  - `安全` → `💞 安全`
  - `品質` → `🏅 品質`
  - `生産` → `🏭 生産`
  - `保全` → `🪛 保全`
  - `環境` → `🌳 環境`
  - `原価` → `💰 原価`
  - `人材育成` → `🏫 人材育成`
  - `安全健康` → `💞 安全健康`
  - `専門技能` → `💪 専門技能`
  - `TPS` → `📊 TPS`
  - `人事` → `👤 人事`
  - `総務` → `🧑‍💻 総務`
- 内容一覧とアップロード待ちキューは **タブ切替表示**
- 内容一覧には **状態列** を表示
  - `変換済`
  - `未変換`
  - `エラー`
  - `変換中`
- 状態は常に最新状況へ追従するよう定期更新
- 内容一覧の **名前 / 状態 / 更新日時 / サイズ** は昇順・降順ソート可能
- 規定値は **更新日時の降順**
- 内容一覧とキューのファイル名左に拡張子アイコン表示
- ドラッグ&ドロップ欄はコンパクト表示
- ドラッグ&ドロップ欄に **追加可能拡張子** と **1ファイル最大100MB** の注意書きを表示
- キューのエラー行に **再試行** / **削除** ボタンを表示（エラー行のみ）

## アップロード制約
- **非対応拡張子は内容一覧で非表示**
- **非対応拡張子はアップロード不可**
- **非対応拡張子はキュー追加不可**
- **100MBを超えるファイルはアップロード不可**（フロント側・サーバー側の両方で拒否）

### app.py
- Flask ルート
- アプリ起動
- キュー/監視インスタンス生成

- フォルダ監視ワーカー
- 自動削除制御

- ondemand画面の骨組み
- ヘッダー / サイドバー / ドロップ領域 / タブ / 内容一覧 / キュー

### static/ondemand.js
- ツリー描画
- API通信
- アップロード
- 内容一覧の状態表示
- ソート
- タブ切替
- キュー表示更新
- 削除操作
- キューエラー行の手動削除・再試行操作

### static/style.css
- 固定レイアウト
- コンパクト表示
- ツリー見た目
- テーブル見た目
- 状態ラベル見た目

- UI と既存機能は変えずに `app.py` を分割済み
- `app.py` はルートと起動処理中心
- 重い共通処理は `ondemand_core.py` へ分離

- 変更時は `CLAUDE.md` も合わせて更新する

## 直近の変更履歴
### Markdown変換の最適化とファイル名リネーム（2026-04-30）
- **最適化分析**: 処理フローを解析した結果、既に最適化されており無駄な処理はほぼなし。時間はDify Chat API応答時間に依存
- **ファイル名リネーム**: Markdownファイル名に変換時刻（yyyymmddhhmmss形式）をプレフィックスとして追加
  - フォーマット: `{変換時刻yyyymmddhhmmss}_{元ファイル名}.md`
  - 例: `20260430143525_1.md`
- **差分判定の改善**: 保存先ディレクトリ内で `*_{元ファイル名}.md` パターンでファイル検索
- **削除処理の改善**: タイムスタンププレフィックス付きファイルを正しく検索・削除

### Difyナレッジ登録の完全撤廃（2026-04-30）
- Markdown変換のみを行い、Difyナレッジへの登録は行わない仕様に変更
- `/api/datasets` エンドポイントを空のリストを返すように変更（Dify接続エラー回避）
- 全Lv5フォルダでアップロード可能に変更（ナレッジ存在チェック不要）
- 差分判定は保存先Markdownファイルの存在チェックのみ
- 削除処理も元ファイル + Markdown のみ（Difyドキュメント削除なし）
- キュー処理も Markdown保存で完了（埋め込み処理なし）
- 状態表示を「変換済/未変換/変換中/エラー」に変更
- エラーメッセージを「Markdown変換API設定が未完了」に変更

### エラーキュー手動操作機能の追加
- キューのエラー行に **再試行** / **削除** ボタンを追加
- 削除: `POST /api/ondemand/queue/<task_id>/delete` — 元ファイル・Markdown を全削除してキューから除去
- 再試行: `POST /api/ondemand/queue/<task_id>/retry` — Markdown のみ削除してリトライカウントをリセットし、キュー最後尾へ再投入
- `ondemand_core.py` に `cleanup_markdown_only()` を追加（元ファイルは残す削除処理）
- `ondemand_queue.py` に `get_error_task_for_action()` / `remove_error_task()` / `retry_error_task()` を追加
- Markdown が存在しない場合の削除はスルー（tolerant 処理）
